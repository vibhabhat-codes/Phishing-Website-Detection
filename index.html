❓ How Attack on Phishing Works
1. Feature extraction

When you submit a URL, the backend extracts two groups of signals:

  • URL-level features (~47) — things like URL length, character entropy, IP-address usage, brand-name spoofing patterns, fake/uncommon TLDs, punycode, use of URL shorteners, and presence of free-hosting domains. These don't require fetching the page.
  • HTML/content features (~23) — pulled by actually requesting the page: number of forms, external vs internal links, inline JavaScript patterns, hidden elements, and whether forms submit to an external domain. If the page can't be fetched (blocked, offline, timeout), the backend falls back to dataset median values instead of zeros, so a single failed fetch doesn't tank the score.
2. The model — a two-stream stacked ensemble

All ~71 features (plus a computed "suspicion score") are imputed, scaled, and passed to a stacking classifier with two base learners feeding a meta-learner:

  • LightGBM — learns URL-pattern-driven phishing signals
  • Random Forest — learns HTML/content-driven signals
  • Logistic Regression (meta-learner) — combines both streams into a final phishing probability
3. Decision logic — dual threshold + overrides

A URL isn't flagged just because the model leans "phishing." It's marked phishing only when both the model's probability and a separately computed suspicion score clear their thresholds — this cuts down on false positives from borderline cases. Three extra rules run first and can force a "phishing" verdict outright: the host being a raw IP address, a known URL shortener, or a fake/high-abuse TLD (e.g. .tk, .xyz). Domains on a small known-legitimate allowlist (e.g. google.com) get a stricter bar before being flagged, to avoid punishing sites the model hasn't seen enough of in training.

4. "Why This Was Flagged" (SHAP)

Each result shows the top features pushing that specific URL's score up or down, computed via SHAP on the LightGBM stream. This explains the URL-pattern learner's reasoning — it's a window into part of the decision, not the full blended verdict (which also includes the HTML stream and the override rules above).

5. Analytics dashboard

Every scan is logged locally to a scans.db SQLite file. The Dashboard tab aggregates that history: total scans, the phishing/legitimate split, a suspicion-score trend over time, and which signals show up most often among flagged URLs.

6. Batch scanning & feedback

The Batch Scan tab runs a CSV or pasted list of URLs through the same pipeline in one go and lets you export the results. The 👍/👎 buttons under each result log whether the verdict was correct, building a record you can use to spot systematic mistakes later.
